  <behavior>
    - new(max_bytes) creates an empty, non-capturing buffer
    - start_capture() begins accumulation, clears any previous data
    - append(data) accumulates bytes when capturing=true
    - append(data) is a no-op when capturing=false
    - append(data) is a no-op when alt_screen=true
    - append(data) stops accumulating after max_bytes reached, but tracks total_seen
    - check_alt_screen(data) detects \x1b[?1049h (enter) and \x1b[?1049l (leave)
    - finish() returns Some(bytes) when capturing was active and not alt-screen
    - finish() returns None when alt_screen was entered during capture
    - finish() returns None when not capturing
    - finish() resets capturing to false
    - total_seen reflects all bytes seen, even those not buffered
  </behavior>

  <action>
    This is integration wiring -- connects OutputBuffer to the PTY read loop and routes captured output to the database.

    Step 1 -- Add CommandOutput variant to AppEvent in `crates/glass_core/src/event.rs`:
    ```rust
    CommandOutput {
        window_id: WindowId,
        output: String,
    },
    ```

    Step 2 -- Integrate OutputBuffer into PTY reader thread in `crates/glass_terminal/src/pty.rs`:
    - Create OutputBuffer in the PTY reader thread (in `pty_read_thread` or equivalent function), initialized with `max_output_capture_kb * 1024` bytes. The max_kb value needs to be passed through spawn_pty. Add a `max_output_capture_kb: u32` parameter to `spawn_pty()` (default 50 if not provided, or pass from config).
    - In the read loop (where `OscScanner.scan(data)` is called), also call `output_buffer.check_alt_screen(data)` and `output_buffer.append(data)`.
    - On `ShellEvent::CommandExecuted` detection: call `output_buffer.start_capture()`.
    - On `ShellEvent::CommandFinished` detection: call `output_buffer.finish()`, process the raw bytes via `glass_history::output::process_output()`, and if Some, send `AppEvent::CommandOutput { window_id, output }` via the event loop proxy.

    Step 3 -- Handle CommandOutput in main event loop in `src/main.rs`:
    - In the AppEvent match arm, add `AppEvent::CommandOutput { window_id, output }`.
    - For now, log the output receipt with tracing::debug. The actual HistoryDb write will need the database to be opened in main -- check if HistoryDb is already accessible. If not, open it lazily (resolve_db_path from cwd, HistoryDb::open) or defer full DB wiring to when the history write path is established.
    - If HistoryDb is accessible: update the most recent command record for this window with the output. This requires tracking the last inserted command_id. If the terminal already inserts CommandRecords on CommandFinished (check main.rs Shell event handling), add the output to that same record. If commands are not yet being inserted into the DB from the terminal (Phase 5 only built the library, not the wiring), then the full write path (insert CommandRecord with output) should be established here.

    Key constraints:
    - OutputBuffer lives entirely in the PTY reader thread -- no mutex needed.
    - Database writes happen on the main thread via AppEvent, never on the PTY thread.
    - The `glass_history` crate must be added as a dependency of `glass_terminal` in Cargo.toml for `process_output` access, OR process_output can be called on the main thread side after receiving raw bytes. Prefer processing on the PTY thread to keep the event payload small (processed String, not raw Vec<u8>).

    Alternative if glass_terminal should not depend on glass_history: Send raw bytes via AppEvent and process on main thread. This keeps the dependency graph cleaner (terminal -> core, history -> core, main -> both). Use this approach if adding glass_history as a dep of glass_terminal creates a circular dependency.
