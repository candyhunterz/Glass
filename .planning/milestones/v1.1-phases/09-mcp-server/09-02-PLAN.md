  <action>
Create `tests/mcp_integration.rs` in the project root (not in glass_mcp crate) so `env!("CARGO_BIN_EXE_glass")` resolves correctly.

1. Write a helper function that:
   - Gets the glass binary path via `env!("CARGO_BIN_EXE_glass")`
   - Spawns `glass mcp serve` as a child process with stdin/stdout piped, working dir set to a temp directory
   - Provides methods to write JSON-RPC requests to stdin and read JSON-RPC responses from stdout

2. Test: `test_mcp_initialize_handshake`
   - Spawn glass mcp serve
   - Write JSON-RPC initialize request to stdin:
     ```json
     {"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"test","version":"0.1.0"}}}
     ```
   - Read response from stdout
   - Assert response contains: `"result"` with `"serverInfo"` containing `"name":"glass-mcp"`, and `"capabilities"` with `"tools"`
   - Send `initialized` notification: `{"jsonrpc":"2.0","method":"notifications/initialized"}`
   - NOTE: JSON-RPC over stdio uses Content-Length headers OR newline-delimited JSON depending on rmcp's transport. Check what rmcp's stdio transport expects. If it uses the LSP-style `Content-Length: N\r\n\r\n{json}` framing, the test must use that format. If it uses newline-delimited, use `{json}\n`.

3. Test: `test_tools_list`
   - After initialize handshake, send `{"jsonrpc":"2.0","id":2,"method":"tools/list","params":{}}`
   - Assert response lists two tools: `glass_history` and `glass_context`
   - Assert each tool has an `inputSchema` with expected properties

4. For reading stdout: use a timeout (5 seconds) to avoid hanging if server doesn't respond. Use a thread with a channel or similar non-blocking approach.

5. Add dev-dependencies to root Cargo.toml if needed:
   ```toml
   [dev-dependencies]
   serde_json = "1.0"
   tempfile = "3"
   ```
