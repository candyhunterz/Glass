# Phase 28: Platform Packaging & CI Release - Research

**Researched:** 2026-03-07
**Domain:** Cross-platform packaging (MSI/DMG/deb) and GitHub Actions release automation
**Confidence:** HIGH

## Summary

This phase packages the Glass terminal emulator binary into platform-native installers (MSI for Windows, DMG for macOS, deb for Linux) and creates a GitHub Actions workflow that builds all three and publishes them as GitHub Release assets on `v*` tag push.

The Rust ecosystem has mature, well-maintained tools for each platform: **cargo-wix** (MSI via WiX Toolset), **cargo-deb** (Debian packages), and manual **app bundle + hdiutil** (macOS DMG). The existing CI workflow already builds on all three platforms (windows-latest, macos-latest, ubuntu-latest) with the correct targets, so the release workflow extends this matrix with packaging steps.

**Primary recommendation:** Use cargo-wix 0.3.9 for MSI, cargo-deb 3.6+ for deb, and a shell script creating an .app bundle + hdiutil for DMG. Trigger release builds on `v*` tag push using `softprops/action-gh-release` to publish assets.

| Tool | Version | Purpose | Why Standard |

### Pattern 1: WiX MSI with Stable UpgradeCode (PKG-01)

**What:** cargo-wix generates a `wix/main.wxs` template. The UpgradeCode GUID must be committed and NEVER changed -- it tells Windows "this is the same product" across upgrades. The PATH entry GUID must also be stable.

**When to use:** Always for MSI installers.

**Example:**

```xml
<!-- wix/main.wxs - Key sections to customize after cargo wix init -->
<Product
    Id='*'
    Name='Glass Terminal'
    UpgradeCode='XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX'
    Manufacturer='Glass'
    Language='1033'
    Codepage='1252'
    Version='!(bind.fileVersion.glass.exe)'>

    <!-- Install to Program Files\Glass Terminal -->
    <Directory Id='TARGETDIR' Name='SourceDir'>
        <Directory Id='ProgramFiles64Folder' Name='PFiles'>
            <Directory Id='APPLICATIONFOLDER' Name='Glass Terminal'>
                <Component Id='binary0' Guid='*'>
                    <File Id='glass.exe'
                        Name='glass.exe'
                        DiskId='1'
                        Source='target\release\glass.exe'
                        KeyPath='yes'/>
                </Component>
                <!-- PATH environment variable entry -->
                <Component Id='Path' Guid='YYYYYYYY-YYYY-YYYY-YYYY-YYYYYYYYYYYY' KeyPath='yes'>
                    <Environment Id='PATH'
                        Name='PATH'
                        Value='[APPLICATIONFOLDER]'
                        Permanent='no'
                        Part='last'
                        Action='set'
                        System='yes'/>
                </Component>
            </Directory>
        </Directory>
    </Directory>

    <Feature Id='Binaries' Title='Application' Level='1'>
        <ComponentRef Id='binary0'/>
        <ComponentRef Id='Path'/>
    </Feature>
</Product>
```

**Critical:** Generate the UpgradeCode GUID once (`cargo wix init` does this), commit `wix/main.wxs`, and never regenerate it.

### Pattern 2: macOS App Bundle + DMG (PKG-02)

**What:** Create a `.app` bundle directory structure with Info.plist, copy the release binary into it, then use `hdiutil` to create a DMG.

**When to use:** Always for macOS distribution outside the App Store.

**Example:**

```bash
#!/bin/bash
# packaging/macos/build-dmg.sh
set -euo pipefail

APP_NAME="Glass"
VERSION="${1:-0.1.0}"
BUNDLE_DIR="target/macos/${APP_NAME}.app"

# Create .app bundle structure
mkdir -p "${BUNDLE_DIR}/Contents/MacOS"
mkdir -p "${BUNDLE_DIR}/Contents/Resources"

# Copy binary
cp target/release/glass "${BUNDLE_DIR}/Contents/MacOS/glass"

# Copy Info.plist
cp packaging/macos/Info.plist "${BUNDLE_DIR}/Contents/Info.plist"

# Create DMG
hdiutil create -volname "${APP_NAME}" \
    -srcfolder "${BUNDLE_DIR}" \
    -ov -format UDZO \
    "target/macos/Glass-${VERSION}-aarch64.dmg"
```

```xml
<!-- packaging/macos/Info.plist -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>CFBundleName</key>
    <string>Glass</string>
    <key>CFBundleDisplayName</key>
    <string>Glass Terminal</string>
    <key>CFBundleIdentifier</key>
    <string>com.glass.terminal</string>
    <key>CFBundleVersion</key>
    <string>0.1.0</string>
    <key>CFBundleShortVersionString</key>
    <string>0.1.0</string>
    <key>CFBundleExecutable</key>
    <string>glass</string>
    <key>CFBundlePackageType</key>
    <string>APPL</string>
    <key>LSMinimumSystemVersion</key>
    <string>11.0</string>
    <key>NSHighResolutionCapable</key>
    <true/>
</dict>
</plist>
```

### Pattern 3: Debian Package via cargo-deb (PKG-03)

**What:** Add `[package.metadata.deb]` to Cargo.toml and run `cargo deb`.

**When to use:** Always for Debian/Ubuntu distribution.

**Example:**

```toml
# In Cargo.toml
[package.metadata.deb]
maintainer = "Glass Contributors"
copyright = "2026, Glass Contributors"
license-file = ["LICENSE", "0"]
depends = "$auto"
extended-description = "A GPU-accelerated terminal emulator that understands command structure."
section = "x11"
priority = "optional"
assets = [
    ["target/release/glass", "usr/bin/", "755"],
    # Shell integration scripts if any are standalone files
]
```

### Pattern 4: Release Workflow (PKG-04)

**What:** GitHub Actions workflow that triggers on `v*` tag push, builds on all three platforms, packages, and uploads to GitHub Release.

**Example:**

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    tags:
      - "v*"

permissions:
  contents: write

env:
  CARGO_TERM_COLOR: always

jobs:
  build-windows:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
        with:
          key: x86_64-pc-windows-msvc
      - name: Build release
        run: cargo build --release
      - name: Install cargo-wix
        run: cargo install cargo-wix
      - name: Build MSI
        run: cargo wix --no-build --nocapture
      - name: Upload MSI
        uses: softprops/action-gh-release@v2
        with:
          files: target/wix/*.msi

  build-macos:
    runs-on: macos-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
        with:
          key: aarch64-apple-darwin
      - name: Build release
        run: cargo build --release
      - name: Create DMG
        run: bash packaging/macos/build-dmg.sh ${{ github.ref_name }}
      - name: Upload DMG
        uses: softprops/action-gh-release@v2
        with:
          files: target/macos/*.dmg

  build-linux:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
        with:
          key: x86_64-unknown-linux-gnu
      - name: Install Linux dependencies
        run: |
          sudo apt-get update
          sudo apt-get install -y libwayland-dev libxkbcommon-dev libx11-dev libxi-dev libxtst-dev
      - name: Build release
        run: cargo build --release
      - name: Install cargo-deb
        run: cargo install cargo-deb
      - name: Build deb
        run: cargo deb --no-build
      - name: Upload deb
        uses: softprops/action-gh-release@v2
        with:
          files: target/debian/*.deb
```

### Anti-Patterns to Avoid

- **Regenerating UpgradeCode:** NEVER run `cargo wix init` again after committing `wix/main.wxs`. A changed UpgradeCode means Windows treats it as a different product entirely -- users get duplicate installs instead of upgrades.
- **Hardcoding version in packaging files:** Extract version from `Cargo.toml` or the git tag. Do not manually edit version strings in `Info.plist`, `main.wxs`, etc.
- **Building inside the packaging step:** Use `--no-build` flag for both cargo-wix and cargo-deb to separate the build step from the packaging step. This avoids double-building and makes the pipeline clearer.
- **Single release job:** Do NOT try to build all platforms in one job. Each platform must build natively on its own runner. `softprops/action-gh-release` handles concurrent uploads to the same release.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| MSI installer | Custom WiX XML from scratch | `cargo wix init` template | Template handles UpgradeCode, PATH, Program Files, x64 correctly |
| Debian packaging | dpkg-deb manually | cargo-deb | Handles control files, maintainer scripts, asset mapping, architecture |
| GitHub Release | Custom API calls | softprops/action-gh-release | Handles release creation, concurrent artifact upload, tag-to-release mapping |
| Build caching | Manual artifact caching | Swatinem/rust-cache | Already used; correctly hashes Cargo.lock and target directory |

**Key insight:** Packaging tools exist specifically because installer formats have complex requirements (MSI has database schemas, deb has control file formats, .app bundles need specific directory layouts). Hand-rolling any of these is error-prone and wastes time.

## Common Pitfalls

### Pitfall 1: UpgradeCode Instability
**What goes wrong:** Each `cargo wix init` generates new GUIDs. If you regenerate, Windows sees a completely different product. Users end up with two entries in Add/Remove Programs.
**Why it happens:** Developers run `cargo wix init` again thinking it's idempotent.
**How to avoid:** Run `cargo wix init` ONCE, commit `wix/main.wxs`, never regenerate. Document the UpgradeCode GUID prominently.
**Warning signs:** Two Glass entries in Windows Programs and Features.

### Pitfall 2: WiX Not in PATH on GitHub Actions
**What goes wrong:** `cargo wix` fails because it cannot find `candle.exe` and `light.exe`.
**Why it happens:** WiX v3.14 is installed on windows-latest but may not be in PATH.
**How to avoid:** Set `WIX` environment variable or add to PATH explicitly in the workflow: `echo "C:\Program Files (x86)\WiX Toolset v3.14\bin" >> $GITHUB_PATH`
**Warning signs:** "candle.exe not found" or "light.exe not found" errors in CI.

### Pitfall 3: macOS Gatekeeper Blocking Unsigned App
**What goes wrong:** Users double-click the .app and get "cannot be opened because the developer cannot be verified."
**Why it happens:** The app is not code-signed or notarized. Code signing is deferred to PKG-F04.
**How to avoid:** Document the workaround: `xattr -cr /Applications/Glass.app` or right-click > Open. Add a note to the GitHub Release description.
**Warning signs:** User reports of "damaged" or "unverified" application.

### Pitfall 4: Linux Dependencies Missing at Runtime
**What goes wrong:** The .deb package installs but Glass crashes because GPU/windowing libraries are missing.
**Why it happens:** `$auto` dependency detection only catches linked libraries, not runtime GPU drivers.
**How to avoid:** Explicitly add runtime dependencies in `[package.metadata.deb]`: at minimum `libwayland-client0, libxkbcommon0` and similar.
**Warning signs:** "Failed to create window" or "No suitable GPU adapter" errors on fresh Ubuntu installs.

### Pitfall 5: Version Mismatch Between Tag and Cargo.toml
**What goes wrong:** The release is tagged v0.2.0 but Cargo.toml says version = "0.1.0". Installer metadata shows wrong version.
**Why it happens:** Developer forgets to bump Cargo.toml version before tagging.
**How to avoid:** Add a CI step that verifies the tag matches `Cargo.toml` version. Fail early if they differ.
**Warning signs:** MSI/deb showing version 0.1.0 when the release is v0.2.0.

### Pitfall 6: softprops/action-gh-release Race Condition
**What goes wrong:** Multiple parallel jobs try to create the same release simultaneously.
**Why it happens:** Three jobs (Windows, macOS, Linux) all use `softprops/action-gh-release`.
**How to avoid:** This is actually handled gracefully by the action -- it creates the release on the first call and subsequent calls just upload additional assets. No workaround needed, but be aware it works this way.
**Warning signs:** None -- this is a non-issue but commonly feared.

### Version tag verification step (CI)

```yaml
- name: Verify version matches tag
  shell: bash
  run: |
    CARGO_VERSION=$(grep '^version' Cargo.toml | head -1 | sed 's/.*"\(.*\)"/\1/')
    TAG_VERSION="${GITHUB_REF_NAME#v}"
    if [ "$CARGO_VERSION" != "$TAG_VERSION" ]; then
      echo "ERROR: Cargo.toml version ($CARGO_VERSION) does not match tag ($TAG_VERSION)"
      exit 1
    fi
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| cargo-bundle for all platforms | Platform-specific tools (cargo-wix, cargo-deb, manual .app) | 2023+ | cargo-bundle poorly maintained; platform-specific tools are more reliable |
| WiX v3 (candle/light) | WiX v4/v5 (wix.exe) | 2023 | cargo-wix still uses WiX v3 which is pre-installed on runners; no migration needed |
| action-gh-release v1 | softprops/action-gh-release v2 | 2024 | v2 uses Node 20, better error handling |

**Deprecated/outdated:**
- `cargo-bundle`: Last meaningful update was 2023; Apple Silicon issues reported. Avoid.
- `actions-rs/cargo`: Unmaintained since 2022. Use direct `cargo` commands instead.
- WiX v3 is in maintenance mode (WiX v4/v5 is current), but cargo-wix still targets v3, which is fine since it's pre-installed on runners.

## Open Questions

1. **Shell integration scripts in installers**
   - What we know: Glass has shell integration scripts (bash, zsh, fish, PowerShell) that are auto-injected via `find_shell_integration()`
   - What's unclear: Are these scripts embedded in the binary or separate files? If separate, they need to be included as installer assets.
   - Recommendation: Check if scripts are embedded at compile time. If separate, add them to MSI/deb assets.

2. **Icon for macOS app bundle and Windows MSI**
   - What we know: macOS .app bundles typically include an .icns file; MSI can embed an icon
   - What's unclear: Does Glass have an icon asset?
   - Recommendation: If no icon exists, use a placeholder or skip for v0.1.0. Icon can be added later.

3. **Linux desktop entry**
   - What we know: A .desktop file enables launching from application menus
   - What's unclear: Whether Glass needs one (it's a terminal, usually launched from another terminal)
   - Recommendation: Include a basic .desktop file in the deb package -- low effort, improves discoverability.
