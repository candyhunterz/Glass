Add messaging methods to the existing `impl CoordinationDb` block in db.rs:

1. `pub fn broadcast(&mut self, from_agent_id: &str, project: &str, msg_type: &str, content: &str) -> Result<u64>`:
   - Begin IMMEDIATE transaction
   - Query all agents in the same project EXCEPT the sender: `SELECT id FROM agents WHERE project = ?1 AND id != ?2`
   - For each recipient, INSERT a message row with `to_agent = recipient_id`, `from_agent = from_agent_id`, `msg_type`, `content`, `created_at = unixepoch()`, `read = 0`
   - This solves the research open question #1: one row per recipient means each agent has independent read tracking for broadcasts
   - Also refresh sender's heartbeat: UPDATE agents SET last_heartbeat = unixepoch() WHERE id = ?
   - Commit. Return the count of message rows inserted.

   IMPORTANT: Do NOT insert a single row with `to_agent = NULL`. Instead, fan out to per-recipient rows. This avoids the shared read flag problem identified in the research. Each recipient has their own message row they can independently mark as read.

2. `pub fn send_message(&mut self, from_agent_id: &str, to_agent_id: &str, msg_type: &str, content: &str) -> Result<i64>`:
   - Begin IMMEDIATE transaction
   - INSERT INTO messages (from_agent, to_agent, msg_type, content, created_at) VALUES (?1, ?2, ?3, ?4, unixepoch())
   - Also refresh sender's heartbeat
   - Commit. Return the message ID (last_insert_rowid).

3. `pub fn read_messages(&mut self, agent_id: &str) -> Result<Vec<Message>>`:
   - Begin IMMEDIATE transaction
   - SELECT messages where `to_agent = ?1 AND read = 0`, JOIN with agents to get from_name:
     ```sql
     SELECT m.id, m.from_agent, a.name, m.to_agent, m.msg_type, m.content, m.created_at
     FROM messages m
     LEFT JOIN agents a ON m.from_agent = a.id
     WHERE m.to_agent = ?1 AND m.read = 0
     ORDER BY m.created_at ASC
     ```
   - Collect messages into Vec&lt;Message&gt;
   - Mark all selected messages as read: `UPDATE messages SET read = 1 WHERE id = ?`
   - Also refresh reader's heartbeat
   - Commit. Return the messages.

   NOTE: Since broadcasts are fanned out to per-recipient rows (each with explicit to_agent), this simple query correctly captures both directed and broadcast messages for the agent. No need for `OR to_agent IS NULL` logic.
