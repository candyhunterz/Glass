       ```rust
       pub struct IpcClient {
           next_id: std::sync::atomic::AtomicU64,
       }
       ```
       - `pub fn new() -> Self` -- just initializes the atomic counter. No eager connection.
       - `pub async fn send_request(&self, method: &str, params: serde_json::Value) -> Result<serde_json::Value, String>` -- connects fresh on each call (handles GUI restart), sends JSON request line, reads JSON response line, returns result or error.

    2. **Platform-specific connect helper**:
       - `#[cfg(unix)] async fn connect() -> Result<impl AsyncRead + AsyncWrite + Unpin, String>` -- connects to `glass_core::ipc::ipc_socket_path()` via `tokio::net::UnixStream`

       - Build JSON: `{"id": N, "method": "...", "params": {...}}`

       - Read response line with 5-second timeout (`tokio::time::timeout`)
       - Parse McpResponse. If `.error` is Some, return Err. Otherwise return `.result`.

This is safe because glass_core has no dependency on glass_mcp (no circular dep). However, glass_core depends on winit which is heavy. ALTERNATIVE: duplicate the 4-line path helpers in glass_mcp to avoid the dependency.     4. **Graceful degradation pattern** (established here, used by all future live tools):
       ```rust
       async fn glass_ping(&self) -> Result<CallToolResult, McpError> {
           let client = match self.ipc_client.as_ref() {
               Some(c) => c,
               None => return Ok(CallToolResult::error(vec![Content::text(
                   "Glass GUI is not running. Live tools require a running Glass window."
               )])),
           };
           match client.send_request("ping", serde_json::json!({})).await {
               Ok(resp) => Ok(CallToolResult::success(vec![Content::text(
                   serde_json::to_string_pretty(&resp).unwrap_or_default()
               )])),
               Err(e) => Ok(CallToolResult::error(vec![Content::text(
                   format!("Failed to communicate with Glass GUI: {}", e)
               )])),
           }
       }
       ```

    This establishes the pattern that ALL future live MCP tools (Phase 36-39) will follow: check ipc_client -> send_request -> handle result/error.
