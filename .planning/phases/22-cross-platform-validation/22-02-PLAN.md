```rust
WindowEvent::ScaleFactorChanged { scale_factor, .. } => {
    tracing::info!("Scale factor changed to {}", scale_factor);
    // Reinitialize font metrics with new scale factor
    let session = self.session_mux.focused_session_mut();
    if let Some(session) = session {
        let (cell_w, cell_h) = self.frame_renderer.update_scale_factor(scale_factor);
        let (width, height) = {
            let size = self.window.as_ref().unwrap().inner_size();
            (size.width, size.height)
        };
        let num_cols = (width as f32 / cell_w).floor() as u16;
        let num_lines = ((height as f32 / cell_h).floor() as u16).saturating_sub(1);
        let _ = session.pty_sender.send(PtyMsg::Resize(WindowSize {
            num_lines,
            num_cols,
            cell_width: cell_w as u16,
            cell_height: cell_h as u16,
        }));
        session.term.lock().resize(alacritty_terminal::term::TermDimensions {
            columns: num_cols as usize,
            screen_lines: num_lines as usize,
        });
    }
}
```

IMPORTANT: Check if `FrameRenderer` has an `update_scale_factor` method. If not, the executor should check what method exists for reinitializing font metrics with a new scale factor. The key behavior is: when scale factor changes, recalculate cell dimensions and resize the terminal grid. If FrameRenderer doesn't support dynamic scale factor updates, add a log warning and skip the resize (document as known limitation for multi-monitor setups).

The core requirement is the event handler arm exists and logs the scale factor change. Full resize support depends on FrameRenderer capabilities.

2. Create `.github/workflows/ci.yml` with a cross-platform build matrix:

```yaml
name: CI

on:
  push:
    branches: [main, master]
  pull_request:
    branches: [main, master]

env:
  CARGO_TERM_COLOR: always

jobs:
  build:
    name: Build (${{ matrix.target }})
    strategy:
      fail-fast: false
      matrix:
        include:
          - os: windows-latest
            target: x86_64-pc-windows-msvc
          - os: macos-latest
            target: aarch64-apple-darwin
          - os: ubuntu-latest
            target: x86_64-unknown-linux-gnu
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
        with:
          key: ${{ matrix.target }}

      - name: Install Linux dependencies
        if: matrix.os == 'ubuntu-latest'
        run: |
          sudo apt-get update
          sudo apt-get install -y libwayland-dev libxkbcommon-dev libx11-dev libxi-dev libxtst-dev

      - name: Build
        run: cargo build --release

      - name: Test
        run: cargo test --workspace

  clippy:
    name: Clippy
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          components: clippy
      - uses: Swatinem/rust-cache@v2
      - run: cargo clippy --workspace -- -D warnings

  fmt:
    name: Format
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          components: rustfmt
      - run: cargo fmt --all -- --check
```

Note: Ubuntu needs wayland/X11 development libraries for winit compilation. The `libwayland-dev libxkbcommon-dev libx11-dev` packages are essential. If the build needs additional dependencies (like libxcb or vulkan headers), the executor should add them based on compilation errors.

3. After creating the CI file, run local cross-compilation checks to validate Plan 01's fixes work. First install targets if not already installed:

Note: `cargo check --target` performs type checking without linking. It will catch compilation errors (missing imports, cfg issues) but not linker errors (which require the target's system libraries). This is sufficient to validate that our cfg-gating and platform-aware code compiles correctly.
