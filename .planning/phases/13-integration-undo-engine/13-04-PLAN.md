<task type="auto">
  <name>Task 2: Gate pre-exec snapshot creation on config.snapshot.enabled</name>
  <files>src/main.rs</files>
  <action>
Close the STOR-03 gap by checking `config.snapshot.enabled` before creating pre-exec snapshots.

The `config` lives on `Processor` (as `self.config`), and the pre-exec snapshot code runs inside `Processor::user_event()` where `self` is available. The snapshot code is inside a block that already has `if let Some(ctx) = self.windows.get_mut(&window_id)`.

1. **In the pre-exec snapshot block (around line 719-746):** Wrap the existing `if let Some(ref store) = ctx.snapshot_store` block with a config check. The logic should be:
   - If `self.config.snapshot` is `None`, snapshots are enabled by default (backward compat -- no [snapshot] section means default behavior).
   - If `self.config.snapshot` is `Some(ref snap)` and `snap.enabled` is `false`, skip pre-exec snapshot creation entirely.

   Replace line 720:
   ```rust
   if let Some(ref store) = ctx.snapshot_store {
   ```

   With:
   ```rust
   let snapshot_enabled = self.config.snapshot.as_ref()
       .map(|s| s.enabled)
       .unwrap_or(true);
   if snapshot_enabled {
     if let Some(ref store) = ctx.snapshot_store {
   ```

   And add the matching closing brace after the existing block closes (after line 746):
   ```rust
     } // if let Some(ref store)
   } // if snapshot_enabled
   ```

   When disabled, log at debug level so there is observability:
   After the `if snapshot_enabled` block, add an else:
   ```rust
   } else {
       tracing::debug!("Pre-exec snapshot skipped: snapshots disabled in config");
   }
   ```

2. **Do NOT gate the Ctrl+Shift+Z handler on config.** Undo should work for snapshots that were already taken (e.g., user enables snapshots, takes some snapshots, then disables -- they should still be able to undo existing snapshots).

3. **Do NOT gate snapshot_store opening on config.** The store is needed for the undo handler and for the watcher drain. Only gate the pre-exec snapshot creation path.

4. **Do NOT gate the FS watcher on config.** The watcher is independent -- it records what changed regardless of whether pre-exec snapshots are taken. This data is useful for Phase 14's UI features.
