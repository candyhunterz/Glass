3. Create `crates/glass_mcp/src/context.rs` with the aggregate query logic:
   - `ContextSummary` struct (Serialize): `command_count: i64`, `failure_count: i64`, `earliest_timestamp: Option<i64>`, `latest_timestamp: Option<i64>`, `recent_directories: Vec<String>`
   - `pub fn build_context_summary(conn: &rusqlite::Connection, after: Option<i64>) -> anyhow::Result<ContextSummary>` that:
     a. Runs aggregate SQL: `SELECT COUNT(*), SUM(CASE WHEN exit_code != 0 THEN 1 ELSE 0 END), MIN(started_at), MAX(finished_at) FROM commands` with optional `WHERE started_at >= ?` for `after`
     b. Runs distinct CWD query: `SELECT DISTINCT cwd FROM commands [WHERE clause] ORDER BY started_at DESC LIMIT 10`
     c. Returns the assembled ContextSummary
   - Use parameterized queries (not string interpolation) for safety. The `after` param can use a Vec<rusqlite::types::Value> + params_from_iter pattern (same as glass_history query.rs).

4. Update `crates/glass_mcp/src/lib.rs`:
   - Declare modules: `pub mod tools;` and `pub mod context;`
   - Create `pub async fn run_mcp_server() -> anyhow::Result<()>` that:
     a. Resolves DB path via `glass_history::resolve_db_path(&std::env::current_dir().unwrap_or_default())`
     b. Creates `GlassServer::new(db_path)`
     c. Calls `server.serve(rmcp::transport::stdio()).await?` and then `service.waiting().await?`
   - NOTE: The exact rmcp serve API may differ between versions. Check rmcp docs/examples if `serve(stdio())` doesn't compile. The pattern is: create transport, create service from handler, await the service.

Do NOT initialize tracing in lib.rs -- that happens in main.rs before calling run_mcp_server().

<task type="auto">
  <name>Task 2: Implement GlassServer with tool handlers and wire main.rs</name>
  <files>crates/glass_mcp/src/tools.rs, src/main.rs</files>

1. Create `crates/glass_mcp/src/tools.rs` with the GlassServer and tool implementations:

   a. Define parameter structs with schemars::JsonSchema + Deserialize:
      - `HistoryParams`: text (Option String), after (Option String), before (Option String), exit_code (Option i32), cwd (Option String), limit (Option usize) -- each field with `#[schemars(description = "...")]`
      - `ContextParams`: after (Option String) -- time filter for context window

   b. Define response struct:
      - `HistoryEntry` (Serialize): command, cwd, exit_code (Option i32), started_at (i64), finished_at (i64), duration_ms (i64), output_preview (Option String)
      - `impl From<CommandRecord> for HistoryEntry` that maps fields, truncating output to 500 chars with "..." suffix

   c. Define `GlassServer` struct:
      ```rust
      #[derive(Clone)]
      pub struct GlassServer {
          tool_router: ToolRouter<Self>,
          db_path: std::path::PathBuf,
      }
      ```

   d. Implement `#[tool_router] impl GlassServer`:
      - `pub fn new(db_path: PathBuf) -> Self` initializing tool_router via `Self::tool_router()`
      - `#[tool(description = "Query Glass terminal command history with filters. Returns commands matching the specified criteria.")]`
        `async fn glass_history(&self, Parameters(params): Parameters<HistoryParams>) -> Result<CallToolResult, McpError>`:
        - Clone db_path
        - In `tokio::task::spawn_blocking`: open HistoryDb, build QueryFilter from params (using `glass_history::query::parse_time()` for after/before strings), call `db.filtered_query(&filter)`
        - Map errors to `McpError::internal_error(msg, None)`
        - Convert results to Vec<HistoryEntry>, serialize with `Content::json()`
        - Return `CallToolResult::success(vec![content])`
      - `#[tool(description = "Get a summary of recent terminal activity including command counts, failure rate, and active directories.")]`
        `async fn glass_context(&self, Parameters(params): Parameters<ContextParams>) -> Result<CallToolResult, McpError>`:
        - Clone db_path
        - In `tokio::task::spawn_blocking`: open HistoryDb, parse `after` string if present via parse_time(), call `context::build_context_summary(db.conn(), after_epoch)`
        - Return `Content::json(&summary)` wrapped in `CallToolResult::success`

   e. Implement `#[tool_handler] impl ServerHandler for GlassServer`:
      - `fn get_info(&self) -> ServerInfo` returning:
        - `protocol_version`: Use the latest ProtocolVersion variant available in the rmcp version being used
        - `capabilities`: `ServerCapabilities::builder().enable_tools().build()`
        - `server_info`: `Implementation::new("glass-mcp", env!("CARGO_PKG_VERSION"))`
        - `instructions`: Some("Glass terminal history and context server. Use glass_history to search commands, glass_context for activity overview.".into())

   NOTE on rmcp imports: The exact import paths may vary by rmcp version. Key types needed:
   - `rmcp::{ServerHandler, ServiceExt, ErrorData as McpError}` (or `rmcp::error::ErrorData`)
   - `rmcp::handler::server::tool::ToolRouter`
   - `rmcp::model::{CallToolResult, Content, Implementation, ProtocolVersion, ServerCapabilities, ServerInfo}`
   - `rmcp::{tool, tool_handler, tool_router}`
   - If `Parameters` extractor: `rmcp::handler::server::tool::Parameters`
   Check rmcp docs if imports don't resolve. The blog examples at https://www.shuttle.dev/blog/2025/07/18/how-to-build-a-stdio-mcp-server-in-rust and https://rup12.net/posts/write-your-mcps-in-rust/ show the full import lists.

2. Update `src/main.rs` -- replace the MCP stub:
   ```rust
   Some(Commands::Mcp { action: McpAction::Serve }) => {
       // MCP server mode: logging to stderr, stdout reserved for JSON-RPC
       tracing_subscriber::fmt()
           .with_env_filter(tracing_subscriber::EnvFilter::from_default_env())
           .with_writer(std::io::stderr)
           .with_ansi(false)
           .init();

       let rt = tokio::runtime::Runtime::new()
           .expect("Failed to create tokio runtime");
       if let Err(e) = rt.block_on(glass_mcp::run_mcp_server()) {
           eprintln!("MCP server error: {}", e);
           std::process::exit(1);
       }
       return;
   }
   ```
   IMPORTANT: This match arm MUST come BEFORE any existing tracing_subscriber::fmt().init() call for the terminal mode. The MCP branch inits its own tracing with stderr writer and returns/exits, so it never reaches the terminal tracing init. If the current main.rs structure has tracing init after the match, this is fine. If tracing init is before the match, move it into the terminal branch only.

   Also add `glass_mcp` to root Cargo.toml [dependencies] if not done in Task 1.
