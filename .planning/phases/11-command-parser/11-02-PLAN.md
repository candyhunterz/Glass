<objective>
Add PowerShell command parsing to the existing command_parser module.

Purpose: Glass runs on Windows where PowerShell is a primary shell. PowerShell uses Verb-Noun cmdlets (Remove-Item, Move-Item) with named parameters (-Path, -LiteralPath) instead of POSIX positional arguments. Without this, destructive PowerShell commands would all fall through to Low confidence, missing pre-exec snapshot opportunities.

Output: PowerShell tokenizer, cmdlet dispatch, and named parameter extraction added to command_parser.rs with unit tests.
</objective>

GREEN phase -- implement PowerShell support:

2. Add `fn is_powershell_cmdlet(cmd: &str) -> bool`:
   - Returns true if the command matches the Verb-Noun pattern: contains a `-` with uppercase letters on both sides (e.g., `Remove-Item`, `Get-Content`)
   - Also returns true for known PowerShell aliases that don't match the pattern: `ri`, `mi`, `ci`, `si`, `gc`, `gci`, `gl`, `gi`, `sc`, `clc`, `sls`
   - Case-insensitive matching

3. Add `fn tokenize_powershell(text: &str) -> Vec<String>`:
   - Simple quote-aware splitter: split on whitespace, respect `"` and `'` quotes
   - No backslash escaping (PowerShell uses backtick, but for file paths this is rare)
   - ~25 lines

4. Add `fn parse_powershell_command(cmd: &str, args: &[String], cwd: &Path) -> ParseResult`:
   - Match on lowercase cmdlet name and aliases (per research Pattern 5 -- PowerShell Command Patterns):
     - Destructive: `remove-item`/`ri`/`del`/`erase`/`rd`/`rmdir`, `move-item`/`mi`/`move`, `copy-item`/`ci`/`copy`, `set-content`/`sc`, `clear-content`/`clc`
     - Read-only: `get-content`/`gc`/`cat`/`type`, `get-childitem`/`gci`/`ls`/`dir`, `get-location`/`gl`/`pwd`, `get-item`/`gi`, `test-path`, `select-string`/`sls`
   - Delegate to `extract_powershell_path_args` for destructive commands
   - For `copy-item`/`move-item`: also extract `-Destination` parameter value

5. Add `fn extract_powershell_path_args(args: &[String], cwd: &Path) -> ParseResult`:
   - Scan args for `-Path` or `-LiteralPath` (case-insensitive) -- next token is the path value
   - If no named path parameter found, treat first positional (non-flag) argument as the path
   - All extracted paths go through `resolve_path()`

6. Add `fn extract_powershell_destination_args(args: &[String], cwd: &Path) -> ParseResult`:
   - Same as path extraction but ALSO looks for `-Destination` parameter
   - Returns both source (-Path) and destination (-Destination) as targets

7. Update `parse_command` to integrate PowerShell:
   - After tokenizing, check if `is_powershell_cmdlet(&tokens[0])` is true
   - If yes, re-tokenize with `tokenize_powershell` and dispatch to `parse_powershell_command`
   - If no, proceed with existing POSIX dispatch
   - This ensures POSIX commands that happen to match aliases (like `del` on Windows cmd) are handled by the correct parser

Run tests -- all PowerShell tests and all existing POSIX tests MUST pass.
